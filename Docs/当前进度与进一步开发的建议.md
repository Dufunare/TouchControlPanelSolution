目前这个项目的架构是这样的，目前基本闭环已经打通
前端qt项目touchcontrolpanelapp,后端静态库项目ControlBackend,两者之间解耦，前端项目依赖于后端静态库项目。共享项目下的devicestate.h是前后端都需要用到的头文件，包含了设备状态的定义。
在Controlbackend中有一个touch设备后端和一个通信后端，前者负责处理touch设备的相关逻辑，后者负责与中转站进行通信。
在这个项目中，我们现在不直接与机械臂通信，而是通过中转站将我们发送的报文转发给机械臂。
touch后端负责处理与touch设备相关的逻辑，目前暴露的接口有初始化设备，启动调度器，终止调度器等等。
通信后端负责与中转站进行通信，包括建立tcp连接，发送报文，接收报文，获取touch坐标并将其发送给中转站等等。
目前的架构设计使得前后端之间的耦合度较低。
前端项目中的mainwindow是需要注意的，这是负责组织呈现各个子widget的主窗口，并且对一些具体的业务逻辑通过信号与槽连线。
前端项目中的devicecontroller是一个重要的类，touch后端和通信后端的接口都在这个类中被调用，负责协调前后端之间的交互。前端发送信号给devicecontroller，devicecontroller根据收到的信号调用相应的后端接口来处理业务逻辑。
widgets目录下的widget都是一些具体的界面组件，目前实现了statuspanel和controlpanel，glcoordinatepanel,分别用于显示状态和信息，控制机械臂的动作，以及显示touch坐标等信息。还有一个videowidget占位，具体逻辑还没有实现。

关于未来，目前有几个方向：
一，实现videowidget，我的建议是获取视频并呈现的部分用c++实现，与touch和通信一样，在后端实现一个video后端，在devicecontroller中引用这个videobackend来实现具体的逻辑，而前端通过devicecontroller来调用videobackend的接口来获取视频数据并呈现。这样可以保持前后端的解耦，同时利用c++在视频处理方面的性能优势。
至于后续的一些图像识别等功能可以后续再考虑是否需要引入python。当然也可以全程用python，在c++中通过调用python接口来实现视频的获取和处理，这样可以利用python在图像处理方面的优势。（但是可能会有延迟方面的问题）
二，优化Opengl的呈现。有一点需要注意，如果需要获取touch的角度信息，可能相应的需要更新touch后端接口和devicestate，因为我目前没有包含这些信息。
三，美化界面，目前的架构已经实现了逻辑与ui的解耦，所以美化界面相对来说比较简单，可以通过修改widget的样式表来实现界面的美化，比如QSS，Qframe等组件的使用，进行布局的调整和圆角化卡片话等。但要注意，美化界面时不要动到业务逻辑的代码，保持前后端的解耦。（不过应该很简单，美化本身不会改变widget的具体实现，只是进行样式的调整）
四，深度优化具体业务逻辑，比如增加touch后端接口，提高鲁棒性，增加错误处理机制，优化通信后端的性能等等。这些优化可以在保持架构不变的前提下进行，进一步提升系统的稳定性和性能。
具体来说，比如我在touch后端添加了一个新的业务接口用来实现一些功能，对应的，我需要在devicecontroller中注册相应的槽函数来处理这个接口的调用，同时在前端的mainwindow中发送相应的信号来触发这个接口的调用。然后在controlpanel中新增一个按钮，并对这个按钮的点击事件进行处理，发送相应的信号来调用这个新的接口。这样就完成了一个新的功能的实现，同时保持了前后端的解耦。另一种情况，比如要新增一个widget，比如一个新的状态显示面板，那么我只需要在前端的mainwindow中添加这个widget，并进行相应的布局调整，同时在devicecontroller中添加相应的接口来提供这个widget需要的数据，这样就完成了一个新的界面组件的添加。

总的来说，目前的架构设计已经实现了前后端的解耦，未来的方向可以在保持架构不变的前提下进行功能的扩展和优化。通过增加新的后端接口，优化现有的业务逻辑，以及美化界面，可以进一步提升系统的功能性和用户体验。同时，在实现新功能时要注意保持前后端的解耦，避免引入过多的依赖关系，以便于后续的维护和扩展。

# 《TouchControlPanel 项目架构总结与协作指南》



## 一、 当前架构概览：前后端解耦模式

目前项目采用了经典的**前后端分离（解耦）架构**。
* **前端（UI层）**：Qt 应用项目 `touchcontrolpanelapp`。
* **后端（业务与设备层）**：C++ 静态库项目 `ControlBackend`。
* **依赖关系**：前端项目单向依赖于后端静态库项目。
* **共享协议**：双方通过共享目录下的 `devicestate.h` 头文件进行数据结构和设备状态的统一约定。

这种架构设计的核心优势在于：**UI 渲染、业务编排与底层设备通信互不干扰。** 这是后续所有功能扩展（如接入视频流、引入图像识别、增加新指令）顺利推进的基础。

### 1. 后端模块 (ControlBackend)
后端目前包含两个核心子模块：
* **Touch 设备后端**：专职处理 Touch 硬件的相关逻辑（如初始化设备、启停调度器等）。
* **通信后端**：负责与中转站的 TCP 交互。目前我们不直接连接机械臂，而是将控制报文、获取到的 Touch 坐标发送给中转站，由中转站进行路由转发。

### 2. 前端模块 (touchcontrolpanelapp)
* **MainWindow**：主窗口，唯一的装配车间。负责组织呈现各个子 Widget，并通过 Qt 的信号与槽机制连接具体的业务逻辑。
* **DeviceController（核心枢纽）**：前后端的“翻译官”与协调者。UI 组件产生的操作信号会发送给它，由它负责调用 Touch 后端或通信后端的具体 C++ 接口；后端的运行状态也由它向上分发给 UI 层。
* **Widgets（界面组件化）**：目前实现了 `StatusPanel`（状态展示）、`ControlPanel`（动作控制）、`GLCoordinatePanel`（Touch 坐标 3D 渲染），以及一个待实现的 `VideoWidget` 占位。

## 二、 未来演进方向与阶段规划

未来的核心目标不是推倒重来，而是**稳住现有边界，完善状态机，标准化接口**。

### 方向 1：视频流接入与 VideoWidget 实现（高优先级）
* **架构设计**：采用 C++ 后端方案。在 `ControlBackend` 中新增 `VideoBackend`（负责拉流、解码、缓冲），前端通过 `DeviceController` 注册/轮询视频帧，`VideoWidget` 仅负责单纯的画面渲染。
* **避坑指南**：**绝对不要**把高频且庞大的视频帧数据塞进现有的 `DeviceState` 中。视频流应走独立的数据通道（回调或队列），`DeviceState` 里只保留轻量级的状态信息（如帧率、丢帧状态、连接状态）。
* **关于 AI 与图像识别**：初期先打通纯视频显示链路。后续如果需要引入 Python 做图像处理，建议采用**进程间通信（IPC）**将模型推理作为独立模块挂载，避免 Python 解释器带来的延迟污染主控制链路。

### 方向 2：业务逻辑深度优化与鲁棒性提升（核心地基）
在开发新功能前，先加固现有业务逻辑：
* **通信稳定性**：强化请求-响应的匹配机制（引入 RequestId），处理 TCP 的粘包/半包分帧问题，并完善断线重连策略与线程回收机制。
* **统一状态机**：Touch 设备、通信链路、机械臂三方都需要有明确的内部状态流转图（如：未连接 -> 连接中 -> 运行中 -> 异常），避免 UI 出现逻辑误判。
* **统一错误模型**：抛弃单纯的字符串报错，定义标准的 `ErrorCode` 体系（如：致命错误、可恢复错误），配合时间戳统一推给前端展示。

### 方向 3：OpenGL 呈现优化与姿态接入
* **数据先行**：如果需要获取 Touch 的角度/姿态信息，首先要在 `DeviceState` 中扩展协议（增加 roll/pitch/yaw 或四元数），明确坐标系约定（世界坐标还是设备坐标，单位是度还是弧度）。
* **增量渲染**：前端 `GLCoordinatePanel` 根据 `DeviceState` 的更新，增量渲染坐标轴、姿态箭头等，保持渲染与业务计算的绝对隔离。

### 方向 4：UI 工程化美化（低风险、见效快）
* **设计令牌（Design Tokens）**：将颜色、圆角、间距、字体等抽象为统一的常量或 QSS 变量。
* **隔离原则**：UI 美化的修改应当严格限制在样式表（QSS）和控件属性内，保持控件的 `objectName` 稳定，**坚决不能触碰或改变底层绑定的业务逻辑槽函数**。

## 三、 团队协作与开发规范（必读）


### 1. 新功能接入的参考工作流（按需裁剪）
当你要增加一个新功能时，请参考以下全链路开发顺序。**请注意：这是一个最完整的标准流程，并非所有功能都需要走完这 5 步。对于无需用户手动干预或无需在前端呈现的纯后端功能，请根据实际情况进行裁剪。**

* **步骤 1：添加后端接口（核心必做）**
    在 `ControlBackend` 的对应模块中实现核心业务逻辑。
* **步骤 2：桥接 Controller（视依赖而定）**
    如果在前端或应用生命周期中需要调度该能力，在前端 `DeviceController` 中注册对应的信号或槽函数来对接该后端接口。纯后端的静默任务可跳过此步。
* **步骤 3：UI 触发源（全链路交互选做）**
    在 `ControlPanel`（或其他 Widget）中新增 UI 控件（如按钮），并在内部处理点击事件，发出纯净的 UI 信号。
* **步骤 4：连线装配（全链路交互选做）**
    在 `MainWindow` 中，将 Widget 发出的 UI 信号 `connect` 到 `DeviceController` 的槽函数上。
* **步骤 5：状态反馈（状态呈现选做）**
    如果该操作的结果需要让用户知晓，确保操作能引起 `DeviceState` 的改变，从而触发 `StatusPanel` 等界面的 UI 更新。

**💡 常见功能类型的裁剪示例：**
* **全链路交互（例：新增“开启机械臂抓取”按钮）**：需要完整走完 1~5 步。
* **后端静默任务（例：新增“TCP 心跳保活机制”）**：只需完成步骤 1（后端实现）。如果断连导致状态变化，只需加上步骤 5（更新状态机），完全不需要 UI 按钮的触发（跳过步骤 3 和 4）。
* **纯数据获取（例：内部轮询 Touch 设备的电量）**：完成步骤 1、2、5。后端获取数据，Controller 调度，更新到状态面板，无需用户手动点击触发。

*** 这样修改后，既保持了架构严谨的分层红线（即使要做全链路，也不能跨层调用），又赋予了协作者足够的灵活度，避免为了写 UI 而写 UI。
### 2. 架构红线（严禁越界）
* **UI 层绝对隔离**：任何 Widget 均不得直接调用 Socket API 或底层设备 API。
* **MainWindow 职责单一**：主窗口只负责 UI 组件的实例化、布局装配和信号槽连线，**禁止在 MainWindow 中编写复杂的业务逻辑（if/else 业务判断）**。
* **防止 Controller 膨胀**：`DeviceController` 很容易随着功能增加变成万能的 God Object。未来视情况，我们会将其按业务拆分为类似 `TouchUseCase`、`CommUseCase` 等专一的调度类。

### 3. 工程习惯建议
* **配置外置**：中转站的 IP、端口、超时时间、发送频率等参数，请放在外部配置文件中，避免硬编码在 C++ 源码里。
* **规范化日志**：建议引入前缀规范，例如 `[时间][模块分类(TCP-TX/Touch/OP)][日志级别] 详细信息`，这将极大降低后续排查 Bug 的血压。
* **分模块提交 (Commit)**：尽量将后端的协议修改、中间层的对接、前端的 UI 调整分开提交，或者在 Commit 信息中清晰标注。
